# AGENTS.md

## Engineering defaults

第一性原理：从原始需求出发，动力不清立刻停止，路径非最优直接纠正。

### 1) 变更策略
- 默认按 clean break 执行，不考虑旧代码兼容。
- 不保留 deprecated API、兼容参数、兼容字段、兼容路径、兼容导入方式。
- 不新增 wrapper、adapter、shim、alias、fallback 来迁就旧实现。
- 新方案一旦确定，直接全量替换旧方案。
- 如果 breaking change 会影响调用方，直接修改调用方或在 `CHANGELOG.md` 中明确迁移方式，不要在代码里留兼容层。

### 2) C/CPP/CUDA 编码风格
- 以可读性、简单性、直接性优先。
- 优先使用短路原则和 early return，避免深层嵌套。
- 优先删除无意义的兼容判断、冗余兜底、重复分支。
- 能直接失败就直接失败，不要静默吞错，不要为了兼容旧输入而做隐式修补。
- 避免 over-engineering；不要为了未来可能需求预埋复杂抽象。

### 3) 修改方式
- 每次任务优先直接修改现有实现，而不是保留旧实现再包一层。
- 能删掉旧代码就删掉，不保留“以后也许有用”的死代码。
- 改接口时，同步更新所有调用点；不要留下新旧接口混用状态。
- 改数据结构时，同步完成全项目替换，不保留兼容转换层。

### 4) CHANGELOG 要求
- 每次完成代码修改后，必须更新 `CHANGELOG.md`。
- `CHANGELOG.md` 至少记录：
  - Changed：这次改了什么
  - Breaking：有哪些破坏性变更
  - Migration：调用方或使用方需要怎样改
- 如果本次没有明显 breaking change，也要写明无兼容性影响。
- 如果仓库里还没有 `CHANGELOG.md`，先创建再更新。

### 5) 输出要求
- 输出使用中文。
- 先给出准备怎么改的简述，再执行修改。
- 修改完成后，汇总：
  - 改了哪些文件
  - 哪些旧逻辑被删除
  - 本次 breaking changes
  - 写入 `CHANGELOG.md` 的内容
  - git commit -m 内容可以怎么写
