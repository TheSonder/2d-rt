# AGENTS.md

## Engineering defaults

第一性原理：从原始需求出发，动力不清立刻停止，路径非最优直接纠正。

### 1) 变更策略
- 默认按 clean break 执行，不考虑旧代码兼容。
- 不保留 deprecated API、兼容参数、兼容字段、兼容路径、兼容导入方式。
- 不新增 wrapper、adapter、shim、alias、fallback 来迁就旧实现。
- 新方案一旦确定，直接全量替换旧方案。
- 如果 breaking change 会影响调用方，直接修改调用方或在 `CHANGELOG.md` 中明确迁移方式，不要在代码里留兼容层。
- 禁止提交处于“过渡态”的实现；不允许新旧接口、新旧字段、新旧逻辑并存。
- 不保留 `TODO remove later`、临时兼容分支、迁移中间层作为后续清理项；本次任务内能完成的迁移必须一次完成。

### 2) 修改前检查
- 开始修改前，先检查影响范围，包括实现、调用点、配置、脚本、测试、文档。
- 需要改接口时，先定位所有调用点，再统一修改，避免只改局部。
- 需要改数据结构时，先确认序列化、反序列化、存储、日志、监控、测试中的相关引用。
- 不允许在未识别主要影响面的情况下直接开始局部修改。

### 3) Python / C++ / CUDA 编码风格
- 以可读性、简单性、直接性优先。
- 优先使用短路原则和 early return，避免深层嵌套。
- 优先删除无意义的兼容判断、冗余兜底、重复分支。
- 能直接失败就直接失败，不要静默吞错，不要为了兼容旧输入而做隐式修补。
- 失败必须显式、可定位、可读；错误信息要能帮助快速排查问题。
- 避免 over-engineering；不要为了未来可能需求预埋复杂抽象。

### 4) 修改方式
- 每次任务优先直接修改现有实现，而不是保留旧实现再包一层。
- 能删掉旧代码就删掉，不保留“以后也许有用”的死代码。
- 改接口时，同步更新所有调用点；不要留下新旧接口混用状态。
- 改数据结构时，同步完成全项目替换，不保留兼容转换层。
- 删除旧逻辑时，同时删除无引用代码、失效注释、过期配置、旧测试、旧文档。
- 不要通过 fallback 或默认分支偷偷维持旧行为；行为变更应明确、可见、可追踪。

### 4A) 代码规模与拆分
- 保持单个文件、单个类、单个函数的职责清晰，避免持续堆积逻辑。
- 当文件同时承担多种职责时，必须拆分；拆分优先按职责边界进行，而不是机械按行数切分。
- 一般建议：
  - Python / C++ / CUDA 单文件超过约 400~800 行时，应评估拆分。
  - 单个函数超过约 40~80 行时，应评估拆分。
  - 单个类过大、状态和行为耦合过多时，应按职责拆分。
- 即使未达到上述行数，只要已经影响可读性、定位效率、修改成本或继续演化，就应立即拆分。
- 不要为了拆分而引入额外抽象、包装层或无意义模块层级；拆分后的结构必须更直接、更清晰。
- 自动生成代码、集中注册文件、表驱动常量定义等可酌情例外，但不得借例外继续堆积无关逻辑。

### 5) 测试与验证
- 每次修改后，必须运行与改动相关的测试或最小验证流程。
- 如果现有测试覆盖旧行为，且本次为 clean break，则直接更新测试，不为旧行为保留兼容测试。
- 如果仓库缺少测试，至少执行最小可复现验证，并在输出中说明验证方式与结果。
- 不允许在未验证主要路径的情况下提交修改。

### 6) 文档同步
- 每次涉及接口、配置、命令、目录结构、使用方式变化时，必须同步更新相关文档。
- 除 `CHANGELOG.md` 外，必要时同步更新 `README.md`、示例代码、脚本说明、配置说明和开发文档。
- 文档内容必须反映修改后的实际行为，不保留旧方案描述。

### 7) CHANGELOG 要求
- 每次完成代码修改后，必须更新 `CHANGELOG.md`。
- 如果仓库里还没有 `CHANGELOG.md`，先创建再更新。
- `CHANGELOG.md` 至少记录：
  - `Changed`：这次改了什么
  - `Breaking`：有哪些破坏性变更
  - `Migration`：调用方或使用方需要怎样改
- 如果本次没有明显 breaking change，也要写明“无兼容性影响”。
- 新记录应追加在顶部，并尽量带上日期或版本号。

### 8) 输出要求
- 输出使用中文。
- 先给出准备怎么改的简述，再执行修改。
- 修改完成后，汇总：
  - 改了哪些文件
  - 哪些旧逻辑被删除
  - 本次 breaking changes
  - 写入 `CHANGELOG.md` 的内容
  - git commit -m 内容可以怎么写
- 如果本次任务仅审阅或提出方案、未实际修改代码，也要明确说明“未改代码”，并列出建议修改点。

### 9） 版本更新
- 代码增加一个记录版本的地方，每次按照更新大小更新版本号
